### front_site — веб-интерфейс инвентаризации

Отдельное веб-приложение на FastAPI, которое повторяет (и развивает) функционал Telegram-бота: авторизация по коду на почту, просмотр активов, отметка инвентаризации, работа с QR и фото.

Файлы бота и основной логики в корне репозитория не трогаем, для сайта пишем отдельный код.

---

### Структура папки

- `app.py` — точка входа FastAPI-приложения, подключение шаблонов и статики.
- `templates/` — HTML-шаблоны:
  - `base.html` — базовый layout (шапка, футер, подключение CSS),
  - `index.html` — стартовая страница (пока без авторизации).
- `static/` — стили и статические ресурсы:
  - `style.css` — минимальный адаптивный стиль в брендовом ключе.
- `logs/` — логи веб-приложения (в git не попадают).

Позже здесь появятся:

- маршруты и шаблоны для авторизации, списка активов, карточек, QR, админ-раздела;
- отдельные модули для работы с A-Tracker, кодами авторизации и декодером QR (копии логики из бота).

---

### Роли «агентов» (кто за что отвечает)

Это не реальные классы, а логические роли, чтобы не путаться в ответственности.

- **Architect / Infra**
  - Структура `front_site` (модули, маршруты, шаблоны, статика, логи).
  - Связь с `config` и `atracker_client`.
  - Запуск под uvicorn, интеграция с IIS (reverse proxy).

- **Auth Agent**
  - Поток авторизации по коду на почту:
    - ввод ФИО/логина/почты,
    - генерация кода,
    - отправка письма,
    - проверка кода и создание сессии.
  - Хранение кодов (in-memory сейчас, задел под Redis/БД).

- **Assets Agent**
  - Список активов сотрудника.
  - Определение статуса инвентаризации (подтверждён / нет).
  - Отметка инвентаризации по одному активу.
  - Карточка актива и работа с файлами (фото).

- **QR Agent**
  - Распознавание QR с фото (через OpenCV / текущий декодер из бота).
  - Извлечение `asset_id` из URL формата `...?ID=123` / `...&ID=123`.
  - Сценарии «свой» / «чужой» актив:
    - обычный пользователь видит сообщение, если актив не его;
    - админ может открыть и работать с чужим активом.

- **Admin Agent**
  - Режим администратора:
    - поиск сотрудника по ФИО/логину/почте,
    - просмотр его активов,
    - отметка за другого пользователя (с логированием).

- **UI/UX Agent**
  - Макеты всех экранов по брендбуку.
  - Адаптив под мобильные и десктоп.
  - Тексты подсказок и ошибок на русском.

- **QA Agent**
  - Чек-листы и сценарии проверки.
  - Smoke-тесты (авторизация, список активов, отметка, QR).

### Итерации и подзадачи

Планируем работу маленькими вертикальными срезами. Каждый шаг должен заканчиваться рабочим сценарием «конец в конец».

#### Итерация 1 — каркас front_site (сделано)

- Создать `front_site/` с базовой структурой: `app.py`, `templates/`, `static/`, `logs/`.
- Настроить FastAPI-приложение с маршрутом `GET /`, который рендерит стартовую страницу.
- Подключить Jinja2 и базовый layout (`base.html`), сделать простую стартовую страницу (`index.html`) без авторизации.
- Добавить минимальный CSS в `static/style.css`.

Результат: сайт открывается по `GET /`, есть шапка/футер и стартовый экран.

#### Итерация 2 — авторизация по коду на почту

- Перенести под `front_site` логику из `auth_by_email.py`:
  - `find_employee_by_input`,
  - `create_code` / `check_code`,
  - `send_code_email`.
- Ввести cookie-сессии:
  - в сессии храним ФИО и почту сотрудника, флаг администратора;
  - TTL кода — 10 минут, сессии — 1–2 часа бездействия.
- Маршруты:
  - `GET /` — форма ввода ФИО / логина / почты,
  - `POST /start-auth` — поиск сотрудника, генерация кода, отправка письма,
  - `GET/POST /enter-code` — ввод и проверка кода, установка сессии.
- UI:
  - экраны ввода, состояния «код отправлен», ошибки (не тот домен, нет сотрудника, неверный/просроченный код).

Результат: пользователь может войти по коду и получить рабочую сессию без списка активов.

#### Итерация 3 — список активов и отметка

- Использовать A-Tracker через `atracker_client.get_assets_by_fio`.
- Логика статуса «подтверждён / нет» такая же, как в боте.
- Маршруты:
  - `GET /assets` — список активов, табы «Все / Не подтверждены / Подтверждены`,
  - `POST /assets/{asset_id}/inventory` — отметка инвентаризации,
  - `GET /no-assets` — экран, если за сотрудником нет техники.
- UI:
  - адаптивная таблица/карточки с кнопками «Подтвердить» и «Подробнее».

Результат: после входа по коду пользователь видит свою технику и может подтверждать активы по одному.

#### Итерация 4 — карточка актива и фото

- `GET /assets/{asset_id}` — карточка актива:
  - вызов `get_asset_info`,
  - блок «Инвентаризация» (статус, дата последней отметки, кнопка «Подтвердить сейчас»),
  - блок «Фото» (список уже загруженных файлов, если отдаёт A-Tracker).
- `POST /assets/{asset_id}/photo`:
  - приём файла `image/*` (телефон или ПК),
  - базовая проверка формата,
  - отправка через `upload_asset_file` в A-Tracker.
- UI:
  - кнопка «Добавить фото»,
  - аккуратное отображение загруженных файлов.

Результат: по каждому активу можно посмотреть детали и прикрепить фото.

#### Итерация 5 — QR по фото

- Вынести в отдельный модуль декодер QR из бота (функция `decode_qr_from_bytes`).
- Маршруты:
  - `GET /assets/qr` — форма загрузки фото QR,
  - `POST /assets/qr` — приём фото, распознавание, поиск `asset_id` в URL.
- Логика:
  - если ID найден и актив принадлежит текущему пользователю — открываем `/assets/{id}`,
  - если актив чужой:
    - обычный пользователь видит сообщение и ФИО владельца,
    - админ может открыть карточку и подтвердить актив.
- UI:
  - кнопка «Сканировать QR» на списке и в карточке актива,
  - подсказки по качеству фото и поведению при ошибках.

Результат: пользователь может сканировать наклейку с телефона и сразу попадать на нужный актив.

#### Итерация 6 — режим администратора

- В `config.ini` (раздел `[web]`) задать `admin_emails` / логины.
- Маршруты:
  - `GET /admin` — дашборд администратора,
  - `GET/POST /admin/search-employee` — поиск сотрудника по ФИО/логину/почте,
  - `GET /admin/assets?employee_id=...` — просмотр и отметка активов выбранного сотрудника.
- UI:
  - явная маркировка, что пользователь в режиме администратора,
  - аккуратные предупреждения при действиях «за другого».
- Логирование:
  - все админ-операции пишем в отдельный лог.

Результат: администраторы могут помогать с инвентаризацией и корректировать данные по другим сотрудникам.

#### Итерация 7 — логи и проверки

- Настроить `logging` для веб-приложения:
  - файл(ы) в `front_site/logs/` с ротацией по размеру/времени,
  - формат: время, пользователь (почта), действие, результат, при необходимости IP/UA.
- Минимальные smoke-тесты:
  - авторизация по коду,
  - получение списка активов,
  - успешная отметка инвентаризации,
  - базовый сценарий QR.

Результат: веб-приложение предсказуемо ведёт себя в продакшене, а проблемные ситуации можно расследовать по логам.

### Развёртывание `front_site` на сервере `atrdbapp.ovp.ru` (10.115.21.77)

Целевая схема:

- сервер: `atrdbapp.ovp.ru` (`10.115.21.77`), на нём уже крутится A‑Tracker;
- путь к проекту: `C:\Users\amelgit\Documents\GitHub\tg_scan` (папка `front_site` внутри);
- публичный адрес сайта для пользователей: `https://myinvent.ovp.ru/`.

1. **Установка Python и создание venv**
   - Установить Python 3.11+ для Windows (с опцией «Add Python to PATH»).
   - На сервере зайти в каталог проекта:
     ```powershell
     cd "C:\Users\amelgit\Documents\GitHub\tg_scan"
     ```
   - Создать и активировать виртуальное окружение, поставить зависимости:
     ```powershell
     py -3 -m venv .venv
     .\.venv\Scripts\activate
     pip install -r requirements.txt
     ```

2. **Запуск uvicorn как службы**
   - Приложение слушает только локальный интерфейс, чтобы наружу ходил только IIS:
     ```powershell
     .\.venv\Scripts\python.exe -m uvicorn front_site.app:app --host 127.0.0.1 --port 8000
     ```
   - Обернуть команду в Windows‑службу (через `nssm install front_site` или `sc.exe`), чтобы uvicorn автоматически стартовал при перезагрузке `atrdbapp.ovp.ru`.
   - Проверить на сервере:
     ```powershell
     curl http://127.0.0.1:8000/
     ```

3. **Настройка IIS как reverse proxy на `myinvent.ovp.ru`**
   - На `atrdbapp.ovp.ru` через «Server Manager»:
     - установить роль `Web Server (IIS)` и модули `URL Rewrite`, `Application Request Routing`.
   - В IIS создать сайт:
     - имя: `myinvent`;
     - привязка: хост `myinvent.ovp.ru`, порт 80/443 (в зависимости от сертификата);
     - физический путь — любая пустая папка (контент всё равно будет проксироваться).
   - В корне сайта настроить правило URL Rewrite:
     - входящие запросы `https://myinvent.ovp.ru/` проксировать на `http://127.0.0.1:8000/`.
   - В ARR включить `Enable proxy` и, при необходимости, `Preserve client IP` (заголовок `X-Forwarded-For`).

4. **Доступ к A‑Tracker**
   - В `config.ini` для сайта `ATRACKER_BASE_URL` указывать как `http://10.115.21.77` (или `http://localhost`, если A‑Tracker слушает на том же хосте и порте 80).
   - С самого `atrdbapp.ovp.ru` проверить доступ:
     ```powershell
     curl http://10.115.21.77/
     ```
   - Если A‑Tracker требует VPN/дополнительный туннель — поднять его на этом же сервере до запуска `front_site`.

5. **Обслуживание и обновления**
   - Обновление кода и зависимостей:
     ```powershell
     cd "C:\Users\amelgit\Documents\GitHub\tg_scan"
     git pull
     .\.venv\Scripts\activate
     pip install -r requirements.txt
     # перезапустить службу front_site через Services / nssm
     ```
   - Логи веб‑приложения:
     - технические: в `front_site/logs/`,
     - аудит действий пользователей: `front_site/logs/audit.log`.
