### Архитектура агентов для tg_scan / front_site

Этот файл нужен, чтобы любая AI-помощь в Cursor сразу понимала, какие есть роли и зоны ответственности в проекте.

---

### Общие правила

- Репозиторий сфокусирован на веб-интерфейсе и общих модулях (`config`, `atracker_client`). Отдельные планировщики (сверка AD, проверка реестра) и Windows exe из этого дерева удалены.
- Веб-интерфейс живёт в `front_site/` и не меняет существующие файлы в корне.
- Общие настройки читаются через `config.py` и `config.ini`. Для веба используется тот же формат INI, но с отдельным разделом `[web]` под свои параметры.

---

### Роли агентов

Эти роли — не классы в коде, а логические зоны ответственности, которыми удобно руководствоваться при разработке.

#### Architect / Infra

- Структура `front_site` (модули, маршруты, шаблоны, статика, логи).
- Подключение `config`, `atracker_client` и других общих модулей.
- Запуск `front_site` под uvicorn, интеграция с IIS (reverse proxy), деплой на Windows Server.

#### Auth Agent

- Весь поток авторизации по коду на почту:
  - ввод ФИО/логина/почты (`/`, `/start-auth`),
  - генерация/проверка кода (`auth_web.py`, `/enter-code`),
  - отправка кода на `@asg.ru`.
- Работа с сессиями (cookie-сессии, поля `user_fio`, `user_email`, `is_admin`, TTL).

#### Assets Agent

- Список активов сотрудника (`/assets`).
- Определение статуса инвентаризации (использует `_is_asset_inventoried`; правила должны совпадать с тем, что ожидает A-Tracker / кастомные сервисы).
- Отметка инвентаризации (`POST /assets/{id}/inventory`).
- Карточка актива (`/assets/{id}`) и загрузка фото (`POST /assets/{id}/photo` → `upload_asset_file` в A-Tracker).

#### QR Agent

- Распознавание QR с фото (логика будет вынесена в отдельный модуль и подключена к маршрутам `/assets/qr`).
- Извлечение `asset_id` из ссылок `...?ID=123` / `...&ID=123`.
- Реакция на «свой» / «чужой» актив:
  - обычные пользователи видят только сообщение и ФИО владельца при чужом активе;
  - администраторы могут открывать карточку и отмечать инвентаризацию.

#### Admin Agent

- Админ-раздел:
  - `/admin` — обзор,
  - `/admin/search-employee` — поиск сотрудника по ФИО/логину/почте,
  - `/admin/assets` — просмотр активов выбранного сотрудника.
- Правила доступа (список админов в `config.ini` → `[web] admin_emails`).
- Понимание, когда админ действует «за другого» и логирование таких операций.

#### UI/UX Agent

- Макеты и логика интерфейса:
  - авторизация, список активов, карточка, QR, админ-раздел.
- Использование брендбука (`assets/reference/Brendbuk_Gajdlajn.pdf`):
  - цвета, шрифты, размеры, отступы.
- Адаптив под мобильные устройства и десктоп.
- Тексты подсказок и ошибок на русском.

#### QA Agent

- Чек-листы для приёмки:
  - авторизация (успех и ошибки),
  - список активов и отметка,
  - карточка и загрузка фото,
  - QR-сценарии,
  - админские операции.
- Smoke-тесты (ручные и, при необходимости, автотесты) для ключевых путей.
