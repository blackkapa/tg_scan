## 1. Что это за проект

`tg_scan` — это система инвентаризации техники с интеграцией в A-Tracker:
- веб-интерфейс `front_site` для сотрудников и админов (основной канал для людей);
- локальные хранилища заявок/перемещений в JSON;
- интеграция через кастомные A-Tracker сервисы.

Планировщики (сверка AD, проверка Excel-реестра) и отдельный exe из репозитория сняты; при необходимости см. `migration_code_backup/` или историю git.

## 2. Главные runtime-компоненты

### 2.1 Web-контур
- Файл: `front_site/app.py` (основной монолит маршрутов)
- Задачи:
  - вход по коду из письма;
  - список активов и карточка;
  - инвентаризация по фото QR;
  - добавление техники (`asset-add`);
  - перемещение техники (`transfer`);
  - админские операции;
  - страница настроек с сохранением в `config.ini`.

### 2.2 Интеграция с A-Tracker
- Файл: `atracker_client.py`
- Канал: `/Api/Login` + `/Api/Service?id=...`
- Используется вебом.

### 2.3 Локальные данные и файлы
- `front_site/data/transfers.json`
- `front_site/data/asset_add_requests.json`
- `front_site/uploads/...`
- `front_site/logs/audit.log`

## 3. Ключевые пользовательские сценарии

### 3.1 Авторизация
1. Пользователь вводит ФИО/логин/почту.
2. Система ищет сотрудника в A-Tracker.
3. Отправляет код на почту (или bypass для whitelist).
4. Проверяет код и создаёт web-session.

### 3.2 Инвентаризация своего актива
1. Открыть `/assets`.
2. Выбрать актив.
3. Загрузить фото QR.
4. Система извлекает `ID=...` и сверяет с `asset_id`.
5. При совпадении вызывается сервис отметки в A-Tracker.

### 3.3 Узнать владельца техники по QR
- Маршрут: `/scan-qr`
- Для обычного пользователя: информирование.
- Для админа: возможны дальнейшие админские действия.

### 3.4 Заявка на добавление техники
1. Пользователь заполняет `/asset-add/start`, добавляет фото.
2. Админ открывает заявку и подтверждает/отклоняет.
3. При подтверждении:
  - если в A-Tracker найден актив по серийному номеру — обновляется существующий;
  - иначе создаётся новый.
4. Уведомление отправляется на почту заявителя.

### 3.5 Перемещение техники
1. Отправитель выбирает активы и создаёт заявку.
2. Отправитель ставит подпись.
3. Получатель подтверждает/отклоняет.
4. Сканы/подписи прикладываются.
5. Админ завершает перемещение через posting-сервис A-Tracker.

## 4. Основные каталоги и назначение

- `front_site/` — web-приложение и его runtime-данные.
- `docs/` — пользовательская и техническая документация.
- `integrations/atracker/services/` — C# сервисы A-Tracker.
- `integrations/atracker/masters/` — XML/мастера A-Tracker.
- `assets/reference/` — эталонные/справочные материалы.

## 5. Настройки, которые критичны для запуска

Файл: `config.ini`

Критичные секции:
- `[atracker]` — URL, логин/пароль, IDs кастомных сервисов;
- `[smtp]` — SMTP для кодов/уведомлений;
- `[email]` — разрешённый домен, админские адреса, bypass;
- `[web]` — `public_base_url`, feature flags (`asset_add_button_enabled`, `transfer_enabled` и др.).

## 6. Маршруты web, которые чаще всего нужны в работе

- `/` — старт входа;
- `/assets` — мои активы;
- `/assets/{asset_id}` — карточка актива;
- `/scan-qr` — определить владельца по QR;
- `/asset-add/start` — подача заявки на добавление техники;
- `/asset-add/{request_id}` — карточка заявки;
- `/transfer/start` — заявка на перемещение;
- `/transfers` и `/transfers/{transfer_id}` — список и карточка перемещений;
- `/admin` — админ-поиск сотрудника;
- `/admin/transfers` — админский список перемещений;
- `/settings` — настройки и аудит.

## 9. Операционные риски, о которых помнить

- Секреты в `config.ini` (не коммитить).
- Session secret и unlock secret в web-коде должны быть заменены на безопасные.
- JSON-хранилища не равны полноценной БД (риск гонок/потерь при росте нагрузки).
- `front_site/app.py` большой и монолитный — правки вносить осторожно.
- Любые изменения в A-Tracker IDs и C# сервисах проверять на тестовом контуре.
