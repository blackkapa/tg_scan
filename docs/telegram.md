Ниже — описание **бывшего** Telegram-бота (как было до переноса в веб), чтобы не потерять контекст по логике и рискам.

## 1) Назначение Telegram-бота (исторически)

Telegram-бот закрывал для сотрудников и админов задачи инвентаризации без веб-интерфейса:
- вход по ФИО/логину/почте + код на email;
- просмотр закреплённых активов;
- проведение инвентаризации по QR-фото;
- отправка заявок в техгруппу (нет в списке, нет QR, несоответствие);
- канал обратной связи через ответы в группе.

Файл: ранее `bot.py` (удалён).

## 2) Точка запуска и жизненный цикл (исторически)

- Бот собирался на aiogram: `Bot`, `Dispatcher`, `MemoryStorage`, long polling.
- Раньше вместе с ботом или отдельным процессом могли работать ежедневные задачи: AD sync (`01:00`), реестр (`07:00`). Эти сценарии из текущего дерева кода убраны; веб — процесс `uvicorn`.

## 3) Зависимости Telegram-контура

## 3.1 Внешние интеграции
- Telegram Bot API (в боте раньше — **aiogram**; сейчас в репозитории бота нет).
- A-Tracker API (`/Api/Login`, `/Api/Service`).
- SMTP (для кода входа на почту).

## 3.2 Внутренние модули
- `atracker_client.py` — все вызовы в A-Tracker.
- `auth_by_email.py` — генерация/проверка кода и отправка письма.
- `config.py` — настройки A-Tracker, SMTP, веб и т.д.

## 3.3 Критичные настройки в `config.ini`
- Секция **`[telegram]`** в проекте не используется (раньше — бот и оповещения реестра).
- **`[atracker]`** — URL, логин/пароль, service IDs.
- **`[smtp]`** + **`[email]`** — параметры отправки и доменная политика.

## 4) Что реализовано в боте (функциональные блоки)

## 4.1 Авторизация пользователя

Состояния:
- `waiting_identifier`
- `waiting_code`

Сценарий:
1. `/start` -> запрос ФИО/логина/почты.
2. По введённому идентификатору ищется сотрудник в A-Tracker.
3. Для обычного пользователя:
   - генерируется 6-значный код на 10 минут;
   - код отправляется на корпоративную почту.
4. После ввода кода — вход в рабочий режим.
5. Для admin ID из `admin_ids`: вход без кода, сразу доступ к активам выбранного сотрудника.

Ограничения:
- коды хранятся в памяти (`_codes`) и пропадают при перезапуске.

## 4.2 Просмотр активов и карточка актива

Состояние:
- `waiting_fio`

Поведение:
- бот грузит активы через `get_assets_by_fio`;
- показывает список с маркерами:
  - `✅` проведена инвентаризация;
  - `❌` не проведена;
- по кнопке актива показывает детали:
  - ID,
  - название,
  - серийный,
  - статус.

## 4.3 Инвентаризация по QR

Состояние:
- `inventory` (и частично обработка в `waiting_fio` по фото/тексту)

Как работает:
1. Пользователь отправляет фото QR (или текст с `ID=`).
2. Бот декодирует QR через OpenCV:
   - `cv2.QRCodeDetector()`
   - несколько вариантов изображения (цвет/gray/resize).
3. Извлекает `asset_id` по шаблону `ID=...`.
4. Проверяет, что актив закреплён за пользователем.
5. Вызывает `mark_inventory(...)` в A-Tracker.
6. Пытается прикрепить фото в карточку актива (`upload_asset_file`).

Если QR «чужой»:
- предлагает повторить фото или завершить шаг.

## 4.4 «У меня есть ещё техника» (заявка в группу)

Состояния:
- `reporting_equipment`
- `reporting_comment`

Сценарий:
1. Пользователь шлёт 1+ фото.
2. Добавляет комментарий (или пропускает).
3. Бот отправляет текст + фото в `report_group_id`.
4. Внутренне запоминает связь сообщения группы с пользователем для ответов.

## 4.5 Сообщить о несоответствии

Состояния:
- `discrepancy_asset`
- `discrepancy_reason`
- `discrepancy_reason_other`
- `discrepancy_extra`

Сценарий:
1. Выбор актива из списка.
2. Выбор причины (из предустановленных или «другой»).
3. Комментарий/фото.
4. Отправка в группу.

Статус по развитию:
- **Планируется перенос в web-приложение как отдельная функция** (аналог Telegram-сценария «несоответствие техники» с выбором актива, причиной, комментарием и фото).

## 4.6 Нет наклейки с QR

Состояние:
- `no_qr_photo`

Сценарий:
1. Из карточки актива кнопка «Нет наклейки с QR».
2. Пользователь отправляет фото техники/серийного номера.
3. Отправка в группу.
4. Дополнительно фото загружаются в карточку актива A-Tracker.

## 4.7 Узнать владельца по QR (информационный режим)

Состояние:
- `identify_device`

Сценарий:
1. Пользователь отправляет QR-фото.
2. Бот определяет `asset_id`.
3. Через `get_asset_info` получает владельца и детали.
4. Показывает, за кем числится актив.

## 4.8 Переписка «группа <-> заявитель»

Важный механизм:
- `report_message_to_user[(chat_id, message_id)] = user_id`
- `user_to_group_thread[user_id] = (group_chat_id, group_message_id)`

Что это даёт:
- ответы системотехника в группе доставляются заявителю в личку;
- заявитель может reply боту, а бот пересылает это reply обратно в группу на исходную заявку.

Через `ATrackerClient`:

- `mark_inventory(...)`

Примечание:
- transfer/asset-add веб-сервисы в основном живут в `front_site`, не в классическом telegram-флоу.

## 7) Ограничения и риски Telegram-контура

- MemoryStorage FSM:
  - при рестарте состояния пользователей теряются.
- In-memory коды авторизации:
  - код теряется при рестарте;
  - не подходит для multi-instance.
- Сильная зависимость от текстового матчинга ФИО/логина.
- Сложная логика reply-маршрутизации в групповых чатах:
  - чувствительна к удалению/изменению контекста сообщений.
- Большой монолитный файл бота (много ответственности в одном месте) — одна из причин переноса в веб.
- Для пользователей Telegram не всегда удобен для «длинных» процессов (asset-add/документы).

## 8) Что уже дублируется и может жить без Telegram

На сегодня веб-контур уже покрывает основную бизнес-ценность:
- авторизация по почтовому коду;
- просмотр и инвентаризация активов;
- QR-сценарии;
- заявки на добавление техники;
- перемещения и админская обработка.

Фактически Telegram остаётся как альтернативный канал взаимодействия и быстрый мессенджерный UX.

## 9) Критерии «готовности к отказу от Telegram» (для следующего шага)

Перед отключением бота должны быть закрыты условия:
- web покрывает все обязательные сценарии пользователей;
- админский и support-процессы из Telegram перенесены в web/email/helpdesk;
- есть замена механизму «ответ из группы -> пользователю»;
- есть коммуникационный план и дата отключения;
- есть fallback-период параллельной работы.

## 10) Что проверить перед обсуждением деактивации

- Кто реально использует Telegram-бота (по логам/метрикам).
- Какие сценарии используются чаще всего:
  - инвентаризация;
  - «нет QR»;
  - «несоответствие»;
  - «у меня есть еще техника».
- Какие из них уже эквивалентно/лучше реализованы в web.
- Какие процессы завязаны на Telegram-группу как на helpdesk-канал.
