# ТЗ: мастер A-Tracker для обработки заявок на добавление техники

Документ для разработчика A-Tracker. Описывает UI-мастер, который работает с заявками `itamRequest` из веба и завершает процесс до статуса `9 (Выполнена)`.

Готовый скрипт для финального перехода мастера:
- `integrations/atracker/masters/ATRACKER_MASTER_AssetAdd_Finalize.cs`
- Готовый XML-каркас мастера:
- `integrations/atracker/masters/ATRACKER_MASTER_AssetAdd_Wizard.xml`

## 1) Назначение

- Веб создает заявку и фото через API.
- Админ работает в A-Tracker через мастер.
- Мастер:
  - показывает данные заявки и вложения;
  - подставляет найденного пользователя (если найден однозначно);
  - позволяет вручную выбрать пользователя при необходимости;
  - завершает заявку и переводит ее в `seStatus=9`.

## 2) Входные данные мастера

- Контекст: `itamRequest.ID` (текущая заявка).
- Ограничение запуска: только заявки `seType=1`, `seStatus=7`.

## 3) Шаги мастера

## Шаг 1. Карточка заявки (readonly)

Поля:
- `sReqNumber`
- `sFullName`
- `sPurpose`
- `sPurchaseItem`
- `lRequesterId` (+ имя)
- Служебные поля автопоиска:
  - `lResolvedUserId` (если есть),
  - `sResolveMode`,
  - `sResolveNote`.

Вложения:
- список документов из контекста `itamRequest`.
- превью фото (если доступно в вашей конфигурации).

Кнопки:
- `Далее`
- `Отмена`

Валидации:
- Заявка существует и `seStatus=7`.
- Вложения присутствуют (минимум 1) — warning, но не hard-block.

## Шаг 2. Пользователь и локация актива

Поля:
- `Пользователь актива` (`lUserId`, lookup itamEmplDept) — обязательно.
- `Локация` (`lLocationId`, lookup itamLocation) — по вашему процессу: обязательно/опционально.
- `Комментарий для актива` — опционально.

Автоподстановка:
- если `lResolvedUserId` заполнен и валиден -> предзаполнить `lUserId`.
- если `sResolveMode=ambiguous|none` -> поле пустое, админ выбирает руками.

Кнопки:
- `Назад`
- `Далее`
- `Отмена`

Валидации:
- `lUserId` обязателен.
- если `lResolvedUserId` есть, но пользователь не найден в lookup -> показать warning и потребовать ручной выбор.

## Шаг 3. Создание/обновление актива

Режим:
- Радиокнопки:
  - `Создать новый актив`
  - `Использовать существующий актив`

### 3.1 Создать новый актив

Поля:
- `sFullName` (из заявки, редактируемо)
- `lCategoryId` / `sCategoryName` (по вашей модели)
- `sSerialNo`
- `sInventNumber`
- `lUserId` (из шага 2)
- `lLocationId` (из шага 2)
- `sComment` (дополнение)

### 3.2 Использовать существующий актив

Поля:
- `lChosenPortfolioId` (lookup itamPortfolio)
- checkbox: `Обновить Пользователь`
- checkbox: `Обновить Локацию`

Правило:
- по согласованной логике по умолчанию обновлять `lUserId` + `lLocationId`.
- остальные поля актива не перетирать автоматически.

Кнопки:
- `Назад`
- `Проверить и завершить`
- `Отмена`

Валидации:
- если выбран режим existing -> `lChosenPortfolioId` обязателен.
- если выбран режим create -> обязательные поля нового актива заполнены.

## Шаг 4. Подтверждение

Экран summary:
- Номер заявки (`sReqNumber`)
- Выбранный пользователь/локация
- Режим (new/existing)
- ID выбранного актива (если existing)

Действие `Завершить`:
- Создать или обновить актив.
- В комментарий актива добавить строку `"[REQ] <sReqNumber>"` без дублирования.
- Обновить заявку:
  - `seStatus = 9`
  - при existing: сохранить `lChosenPortfolioId`
  - опционально `dtProcessed`, `lProcessedBy`.

Кнопки:
- `Назад`
- `Завершить`
- `Отмена`

Валидации:
- перед финалом повторно проверить `seStatus=7` (защита от гонки).
- при ошибке записи актива — не переводить заявку в `9`.

## 4) Статусы и переходы

- Вход в мастер: только `7 (Зарегистрирована/На проверке)`.
- Выход успешный: `9 (Выполнена)`.
- Ручное отклонение (если требуется вашей политикой) — отдельной кнопкой вне данного мастера или отдельным режимом.

## 5) Логирование (рекомендуется)

Логировать в журнал A-Tracker:
- старт мастера (`requestId`, `operatorId`);
- авто-подстановку пользователя (`resolveMode`, `resolvedUserId`);
- финал (`newAssetId` или `chosenPortfolioId`, `status=9`);
- ошибки финализации.

## 6) Нефункциональные требования

- Все шаги мастера должны работать без зависимости от веба.
- Повторный запуск мастера на уже `seStatus=9` должен быть заблокирован.
- Ошибки API/БД показывать с понятным сообщением администратору.
